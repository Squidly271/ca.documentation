Community Applications (Apps Tab) Module

Overview

 Community Applications is a plugin designed to make finding, installing, and managing docker and plugin applications. When using this plugin, there is no longer any need to use the Repositories section of the Docker Tab From top to bottom will explain all aspects of Community Applications starting from the basic and going to the more advanced options. All items within this plugin have tooltips enabled to further guide you in operation. For a help for the settings for this plugin, load the settings page and press Help. Each item will have associated help text displayed. After installation, a new tab will appear on unRaid's main menu called "Apps". Additionally, there will be a section within the Settings tab called Community Applications which has some extra utilities to customize the operation of this plugin.

Categories

 To begin with, at the top of the page are the Category navigation buttons. Clicking these buttons will display the recommended applications within the appropriate category. Some of the categories (Network Services, Media Applications, and Media Servers) also have "sub-categories" to choose from.

View Modes

 These buttons change how the results are displayed. Either displayed in a table, icons, or a combination of the two (Icon Details). Additionally, you can sort the results by Name, Repository, Author, or Date Changed

    Resource Monitor
    The resource monitor will display some statistics on the currently running docker applications (% CPU, % Memory, Network Utilization). Additionally, it will also calculate the size of the various application's appData.
    It also features cAdvisor integration. If the cAdvisor application is not currently installed, a direct link to install it will be present (highly recommended to install it). Once installed, clicking on each application's icon will take you directly to cAdvisor's page for that application which will display even more useful information about the application.

    General Settings
    Within unRaid's Settings tab, there is a section called Community Applications. A program called General Settings allows you to fine tune the operation of Community Applications as detailed below:

    Time to display applications as new / updated
    This option dictates the amount of time that a new or updated application will be flagged as new and displayed under the New / Updated category. Defaults to 3 Months
    Initial View Mode
    This option sets the initial view mode when you first load the plugin. Defaults to Icons
    Favourite Repository
    This option allows you to set a "favourite" repository. If set, then applications from that repository will display before all others. Defaults to None
    Icon Size
    This option only applies to view modes Icons and Icon Details. It sets the displaying of the icons to be either large or small (Defaults to Large)
    Number of Columns
    This option only applies to Icon mode. It will set the number of columns of applications displayed per line. Useful on mobile devices. Defaults to 5
    Group Beta and Private Apps into Super Categories
    This option separates Private and Beta applications into their own sections on any displayed result
    Open Add / Edit / Settings in new Tab
    This option determines whether or not a new tab will open when you add or edit or go to an app's settings. Defaults to the same tab.
    Hide Incompatible Applications
    If CA thinks that an application is incompatible with the version of unRaid that you are using, by default it will not display the application.
    Display Random App Of The Day
    Enables/disables the displaying of 2 random apps of the day when entering the Apps tab
    Enable additional search results from dockerHub
    This option enables the plugin to perform additional searches from dockerHub when looking for an application. See above (Advanced - dockerHub integration for more details) Default: No

    CA Auto Update Module

    Auto Update Settings

     Within the Community Applications section of Settings there is a program called "Auto Update Settings". This program will allow your unRaid server to automatically update any or all of the plugin applications currently installed to the latest versions as they become available without any user intervention.
    To utilize this option, then you must first configure the dynamix UI to check for plugin settings. This is done via Settings - Notification Settings - Plugin Version Notification Set this value to be whatever you desire (I recommend at least once per day). Community Applications will then automatically install any available updates. Note that you should still keep the Notifcation method of Browser / Email / Agents enabled simply to let unRaid inform you of any updates to the OS itself. However if you keep Browser notifications active, you will still have to acknowledge the alert even if the plugin has already been updated by Community Applications.
    unRaid's syslog will have some additional entries as Community Applications installs or skips (if the plugin is set to not autoupdate).
    Send Notifications On Update
    Defaults to Yes. This will send a notification whenever an application has been updated
    Delay In Days Before Updating Applications
    Set this to the number of days after an application has been updated before autoupdating the application. (Setting this to 0 will update the applications the same day they are released) Defaults to 3 days.
    Note: Community Applications will NOT automatically install any unRaid OS updates.


    CA Appdata Backup/Restore Module

    Backup Appdata Settings

    Within the Community Applications section of Settings there is a program called "Backup Appdata Settings". This module will allow you to either manually or on a schedule backup your application's appdata folders from a cache-only share to the parity protected array.
    The settings are as follows:
    Appdata Share (Source)
    This is the cache-only share where all of your docker appdata is stored (usually /mnt/cache/appdata). Note that if your appdata folder is not listed here, then this module will not function for you.
    Save to Disk#
    This sets which disk on your array to save the backup to (see notes below)
    Destination Share
    This sets which share (and subfolder if chosen) to store the backup to (see notes below)
    Excluded Folders
    Use the excluded folder browser to select which folders within your appdata share to not include in the backup (ie: if you are also storing VM images within the appdata folder)
    Backup USB Flash Drive
    This will backup up your USB flash drive. Settings are (Same folder as appdata, separate folder, don't backup). An additional file is automatically added to the config folder of the backup (and on the flash drive) called DISK ASSIGNMENTS.txt) This contains all of the drive assignments (Parity, cache, disk1, etc) current as of the backup time. /boot/config/super.dat is renamed just in case of a hardware change happening inbetween backups (ie: the parity disk at the backup was exchanged and the original placed into the array) which can cause data loss.
    Also, note that in the case of the destination being placed on the array (pretty much guaranteed in the case of same folder as appdata), then in the event of a complete flash failure, you cannot gain access to any of the backed up settings until you assign all of the disks back. For the ultimate in protection, use the unassigned devices plugin and set the destination to be a SMB share from your desktop computer.
    Notification Settings
    This determines under what circumstances a notification (set up via Settings - Notification Settings) is sent: (Disabled, Start And Stop, Completion Only, Errors Only)
    Path To Custom Stop / Start Script These two options allow you to run a custom script to run prior to the backup (and after the backup is completed) in order to perform any maintenance tasks, shut down plugins (if you want them backed up), etc
    Rsync Options
    This entry will allow you to add any custom options to the rsync command that the backup operation uses. If you don't know what you're doing, don't touch this entry!
    Run Scripts in place of or in addition to
    Setting this to "in place of" effectively disabled the backup command, and allows the custom scripts to perform the actual backup. Note that CA will still automatically stop and start the docker applications for you.
    Scheduled Backup Frequency
    Set these options to determine how often to run a scheduled backup operation (or include a custom CRON entry to manually make the schedule yourself)
    Put backups into separate dated subfolders?
    Setting this will create a separate folder (date and time) each time you run the backup.
    Delete backups after how many days?
    This is available when creating separate folders for each backup. It will automatically delete backups that are older than the number of days you specify. After a successful backup, then the old backups will be deleted. In the event of a failed backup, no old backup sets will be deleted.
    Attempt faster rsyncs?
    This option is available when creating separate foldersfor each backup. When it is set to "no (default)", the backup (rsync) command will have to copy ALL of the files over. If you set it to "yes", then prior to copying, one of the backups which are slated to be deleted will be renamed to be the same date and time as the backup that is about to run. This will allow the rsync command to only have to copy the files which are changed over. This can result in significantly faster backup times. The downside is that in the event of the backup failing (for whatever reason), you will have lost the previously made backup.
    NOTES:
    All files within your chosen destination folder will be deleted during a backup process
    When using unRaid 6.1.x, destinations have to be sent to a share and to a disk. The reasons for this is due to hardlinks (which is a subject unto itself), but the net result is that the only way to guarantee a successful backup / restore is to send the backup to a disk share.
    Using 6.2+, destinations can be set to a user share (select user as the disk #), as 6.2 properly supports hardlinks on user shares. The share you select however should be set to not use the cache drive however.

    Restoring your appdata

    In the case of a cache drive failure, the steps required to restore your docker apps back to operational status is as follows:
  • Replace the cache drive
  • Recreate the docker.img file via Settings - Docker Settings
  • Go to Settings - CA section - Backup/Restore
  • Select the restore tab
  • Verify the source and destination which appear and select a backup set to restore
  • After its finished, ideally you should also restart your server so that all changes made will take effect 100%

    • The restore module will then copy the backup set back to your appdata share (and automatically set it to be cache-only). Additionally, it will also delete any files which may have somehow been written to the array during the cache drive failure.

    CA Cleanup Appdata Module

    Cleanup Appdata

    If you have been using docker for a while, and installing and uninstalling various applications, your appdata share probably has a bunch of old folders and files which are no longer needed, and hard to delete due to their permission settings. The Cleanup Appdata modules (Settings -> Community Applications section) will determine which appdata folders (based upon your previously installed docker applications) are no longer needed and then prompt you to delete them. This module will NOT touch any appdata folder which may be used for VMs, plugins, etc

    Other

    Other Notes

    Community Applications (and its predecessor Community Repositories) introduced a number of additional XML tags in order to perform it's magic. Any application / template author and/or maintainer should check out the list of schema additions here: http://lime-technology.com/forum/index.php?topic=40299.0
    Depending upon which browser you are using, there may be a slight delay (~1-2 seconds) in displaying the results (in particular, All Applications using Table mode). Some browsers are faster than others are sorting out all the displayed information. In my testing, on a PC, Chrome is by far the fastest at rendering the results. Note that generally whenever CA searches or re-renders the display, a spinning ring will appear. When that ring stops spinning, CA has finished, and any delays from that point onwards are 100% entirely due to your browser
